## PR 2: Inbox Data Layer

Branch: `feat/pr2-inbox-data`

Files touched:
- `src/db/client.ts` ‚Äî lazy-init Drizzle via Proxy so `next build` works without DATABASE_URL.
- `src/lib/queries.ts` (new) ‚Äî `getCreatorForClerkUser`, `getConversations`, `getConversationDetail`, `getMessages`. Drizzle, no raw SQL except correlated subqueries for last-message preview.
- `app/inbox/page.tsx` ‚Äî replaced 879-line client mockup with Server Component that resolves Clerk user ‚Üí creator ‚Üí conversations, passes into `<InboxShell>`. `export const dynamic = "force-dynamic"`.
- `app/inbox/[conversationId]/page.tsx` (new) ‚Äî deep-linkable route; loads conversation detail + messages; defense-in-depth check that conversation belongs to the resolved creator.
- `src/components/inbox/inbox-shell.tsx` (new) ‚Äî Client Component, ~600 lines, keeps Liquid Glass aesthetic, prop-driven only. Empty states for no conversations / no selection / no messages / no profile.

Key decisions:
- DB client switched to Proxy-based lazy init. The eager `neon(process.env.DATABASE_URL!)` call broke build-time page-data collection; the Proxy preserves the singleton export shape so no callers changed.
- Both inbox routes are `dynamic = "force-dynamic"`. Static prerendering would leak one creator's queue to another.
- Existing orphan files in `src/components/inbox/` (conversation-list.tsx, message-view.tsx, etc.) left untouched; nothing imports them and they don't trip the verifier's banned-pattern regex. Deletion deferred to a future cleanup PR.

Open questions:
- Should the `[conversationId]` route stream messages via Suspense, or wait until SSE arrives in Phase 2? Today everything resolves before the shell renders.
- The 50-message default cap on `getMessages` is arbitrary. Phase 2 adds scroll-to-load-older once we have real volume.

## PR 3: Security Perimeter

Branch: `feat/pr3-security`

Files touched:
- `src/env.mjs` (new) ‚Äî `@t3-oss/env-nextjs` schema with required/optional split; skipValidation during build.
- `src/lib/rate-limit.ts` (new) ‚Äî in-process token bucket; `consume(key, cfg)` returns `{success, remaining, reset}`; `RATE_LIMITS` presets for webhook / aiProcessing / authRoute.
- `src/lib/audit-log.ts` (new) ‚Äî typed `writeAuditLog` that redacts snapshots and never throws; writes to `policy_events` today, Phase 2 splits to `audit_logs`.
- `middleware.ts` (new, repo root) ‚Äî `clerkMiddleware` gating `/inbox`, `/dashboard`, `/settings`, `/conversations` with conditional fallback when Clerk keys absent; rate-limit headers on every response.
- `app/api/webhook/fansly/route.ts` (new) ‚Äî HMAC-SHA256 verify + `timingSafeEqual`; `x-connector-id` header ‚Üí `connector_accounts` lookup; per-IP rate limit; audit_log on accept.
- `src/app/api/webhook/fansly/route.ts` deleted (was unreachable ‚Äî `app/` takes precedence in Next 15).

Key decisions:
- In-process rate limiter for v1. Upstash is the right tool at scale; documented as setup action in PHASE_1_COMPLETE.
- Clerk middleware conditional ‚Äî without keys, falls through to pass-through with rate limits intact. Lets sandbox mode work.
- Webhook handler fails closed: missing `FANSLY_WEBHOOK_SECRET` ‚Üí 503. No unsigned traffic accepted ever.
- `src/env.mjs` uses `skipValidation` during build because `next build` doesn't have real secrets in CI / Railway preview.

Open questions:
- Should rate-limit buckets share state across Railway replicas eventually (Upstash) or per-replica suffices for current scale?
- The webhook `x-connector-id` header contract is internal; what does Fansly actually send? Verifying with their integration docs is the unblock; for now we trust the signature + require the header.
- Should `audit_log` rows also stream out to Axiom for the Phase-2 audit explorer? Decision deferred.
