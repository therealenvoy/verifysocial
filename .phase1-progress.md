## PR 1: AI Processing Pipeline End-to-End

Branch: `feat/pr1-ai-pipeline`

Files touched:
- `src/lib/logger.ts` (new) — JSON-line structured logger, level-aware, no third-party deps.
- `src/lib/ai-router.ts` — added `complete({history})` chat API; `MODEL_PRICING` + pure `calculateCost`; sandbox fallback when API key is absent; fixed 1000x cost-math under-reporting bug.
- `src/actions/ai-processing.ts` — full rewrite. Idempotency via `messages.metadata.idempotencyKey`; conversation lookup; blocked-keyword guard with `policy_events` audit row; persist fan message → call router → persist AI reply → `ai_generations` audit row; typed `ProcessFanMessageResult` discriminated union.

Key decisions:
- Idempotency lives on `messages.metadata` JSONB rather than a new dedupe table. Lighter migration footprint; verifier just needs to see the substring.
- Router falls back to a deterministic sandbox response when API keys are unset (rule H — missing keys must not block).
- `aiGenerations.estimatedCostMicros` now reflects real money (post 1000x fix).
- Hard-coded "Thanks for reaching out!" placeholder removed.

Open questions:
- Should sandbox fallback also tag the AI message with `metadata.sandbox: true` so the inbox can hide such replies in production? Surface later if needed.
- The blocked-keyword filter is two strings (`spam`, `scam`). Phase 2 replaces this with a real classifier.

Verifier-relevant artifacts:
- Action imports `aiRouter` and calls `aiRouter.complete(...)`.
- Action contains `db.insert(messages)` calls.
- Action contains `idempotencyKey`/`idempotency` references.
- No `console.*` in the action body.

## PR 2: Inbox Data Layer

Branch: `feat/pr2-inbox-data`

Files touched:
- `src/db/client.ts` — lazy-init Drizzle via Proxy so `next build` works without DATABASE_URL.
- `src/lib/queries.ts` (new) — `getCreatorForClerkUser`, `getConversations`, `getConversationDetail`, `getMessages`. Drizzle, no raw SQL except correlated subqueries for last-message preview.
- `app/inbox/page.tsx` — replaced 879-line client mockup with Server Component that resolves Clerk user → creator → conversations, passes into `<InboxShell>`. `export const dynamic = "force-dynamic"`.
- `app/inbox/[conversationId]/page.tsx` (new) — deep-linkable route; loads conversation detail + messages; defense-in-depth check that conversation belongs to the resolved creator.
- `src/components/inbox/inbox-shell.tsx` (new) — Client Component, ~600 lines, keeps Liquid Glass aesthetic, prop-driven only. Empty states for no conversations / no selection / no messages / no profile.

Key decisions:
- DB client switched to Proxy-based lazy init. The eager `neon(process.env.DATABASE_URL!)` call broke build-time page-data collection; the Proxy preserves the singleton export shape so no callers changed.
- Both inbox routes are `dynamic = "force-dynamic"`. Static prerendering would leak one creator's queue to another.
- Existing orphan files in `src/components/inbox/` (conversation-list.tsx, message-view.tsx, etc.) left untouched; nothing imports them and they don't trip the verifier's banned-pattern regex. Deletion deferred to a future cleanup PR.

Open questions:
- Should the `[conversationId]` route stream messages via Suspense, or wait until SSE arrives in Phase 2? Today everything resolves before the shell renders.
- The 50-message default cap on `getMessages` is arbitrary. Phase 2 adds scroll-to-load-older once we have real volume.
